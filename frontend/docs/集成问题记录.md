# Frontend 集成问题记录

本文档记录将 `client` 项目集成到 `frontend/pages/apps/image-processing` 过程中遇到的问题及解决方案。

## 1. dayjs ESM 模块兼容性问题

### 问题描述

```
500
The requested module '/_nuxt/@fs/D:/AI/python-code/monorepo-image-processing/node_modules/.pnpm/dayjs@1.11.19/node_modules/dayjs/dayjs.min.js?v=xxx' does not provide an export named 'default'
```

### 原因

`@element-plus/nuxt` 模块在 SSR 模式下使用 dayjs，但 dayjs 的 CJS 模块与 Vite ESM 不兼容。

### 解决方案

**不使用 `@element-plus/nuxt` 模块**，改用 `.client.ts` 插件方式手动注册 Element Plus：

1. 创建 `frontend/plugins/element-plus.client.ts`：

```typescript
import ElementPlus from "element-plus";
import zhCn from "element-plus/es/locale/lang/zh-cn";
import * as ElementPlusIconsVue from "@element-plus/icons-vue";
import "element-plus/dist/index.css";

export default defineNuxtPlugin((nuxtApp) => {
  nuxtApp.vueApp.use(ElementPlus, { locale: zhCn });
  for (const [key, component] of Object.entries(ElementPlusIconsVue)) {
    nuxtApp.vueApp.component(key, component);
  }
});
```

2. 从 `nuxt.config.ts` 中移除：

```typescript
// 移除这些配置
modules: ["@element-plus/nuxt"],
elementPlus: { importStyle: "scss" },
```

## 2. 组件自动导入命名问题

### 问题描述

```
[Vue warn]: Failed to resolve component: ImageUploader
[Vue warn]: Failed to resolve component: SimilarImageGrid
```

### 原因

Nuxt 对 `components/` 目录下的组件自动命名规则：

- `components/image-processing/ImageUploader.vue` → `ImageProcessingImageUploader`
- `components/image-processing/SimilarImageGrid.vue` → `ImageProcessingSimilarImageGrid`

但原 `client` 项目的组件内部使用了短名称引用。

### 解决方案

修改组件内部引用，使用完整的 Nuxt 组件名：

**ProcessSection.vue**:

```diff
- <ImageUploader ... />
+ <ImageProcessingImageUploader ... />
```

**SimilarSection.vue**:

```diff
- <SimilarImageGrid ... />
+ <ImageProcessingSimilarImageGrid ... />
```

## 4. 静态资源路径问题

### 问题描述

```
Failed to resolve import "/pictures/智图寻宝.png" from "components/image-processing/PageHeader.vue"
```

### 原因

组件引用 `/pictures/xxx.png`，但图片在 `client/public/pictures/` 而非 `frontend/public/pictures/`。

### 解决方案

复制静态资源：

```powershell
Copy-Item "client/public/pictures/*" "frontend/public/pictures/" -Recurse
```

## 5. CSS 样式丢失

### 问题描述

页面样式丢失，Element Plus 组件没有正确渲染。

### 原因

1. Element Plus plugin 没有正确加载
2. client 项目的 CSS 变量系统没有导入

### 解决方案

1. 确保 `plugins/element-plus.client.ts` 存在且导入了 CSS：

```typescript
import "element-plus/dist/index.css";
```

2. 在 `nuxt.config.ts` 中添加 client 的 CSS：

```typescript
css: ["~/assets/styles/main.scss", "~/assets/styles/client.css"],
```

## 6. Python 模块命名和导入问题

### 问题描述

```
ModuleNotFoundError: No module named 'apps.image_processing'
ModuleNotFoundError: No module named 'config'
```

### 原因

1. 目录名 `image-processing`（连字符）无法作为 Python 模块名
2. `data.py`、`train.py` 等文件中使用绝对导入 `from config import ...`

### 解决方案

**1. 重命名目录**（连字符改下划线）：

```powershell
Rename-Item "backend/apps/image-processing" "image_processing"
```

**2. 修复模块导入**（在被包导入时使用相对导入）：

`data.py` 文件：

```diff
- from config import IMG_PATH, IMG_SIZE, ...
+ from .config import IMG_PATH, IMG_SIZE, ...
```

## 7. wait-on HEAD 请求问题

### 问题描述

```
[server] INFO:     127.0.0.1:6754 - "HEAD / HTTP/1.1" 405 Method Not Allowed
```

`wait-on` 使用 HEAD 请求检测后端是否启动，但后端 `/` 路由只支持 GET 方法。

### 解决方案

修改 `backend/app.py`，使用 `api_route` 同时支持 GET 和 HEAD：

```diff
- @app.get("/")
+ @app.api_route("/", methods=["GET", "HEAD"])
  async def root():
      ...

- @app.get("/health")
+ @app.api_route("/health", methods=["GET", "HEAD"])
  async def health():
      ...
```

## 8. concurrently 脚本过早退出

### 问题描述

```
[open] wait-on http://localhost:3000 && start http://localhost:3000 exited with code 0
--> Sending SIGTERM to other processes..
```

`start` 命令打开浏览器后立即退出（code 0），触发 `-k` 参数终止所有其他进程。

### 解决方案

修改 `package.json` 中的 `dev` 脚本：

```diff
- "dev": "concurrently -k ... \"wait-on http://localhost:3000 && start http://localhost:3000\""
+ "dev": "concurrently --kill-others-on-fail ... \"pnpm dev:server\" \"wait-on http://127.0.0.1:9000 && pnpm dev:client\""
```

移除自动打开浏览器的命令，使用 `--kill-others-on-fail` 替代 `-k`。

## 9. 模型路径错误

### 问题描述

```
去噪处理失败: 模型未初始化，请先调用 init_model()
```

后端启动时提示模型不存在。

### 原因

`app.py` 中的模型路径指向旧的目录结构：

```python
OLD_MODELS_DIR = ROOT_DIR / "packages" / "server" / "python"
```

但模型实际在 `backend/storage/image-processing/checkpoint/` 目录。

### 解决方案

修改 `backend/app.py` 中的模型路径：

```diff
- OLD_MODELS_DIR = ROOT_DIR / "packages" / "server" / "python"
- denoiser_path = OLD_MODELS_DIR / "image_denoising" / "denoiser.pt"
+ STORAGE_DIR = BASE_DIR / "storage" / "image-processing" / "checkpoint"
+ denoiser_path = STORAGE_DIR / "denoiser.pt"
```

### 问题描述

相似商品推荐返回的图片 URL `/dataset/10000.jpg` 无法访问，图片无法显示。

`app.py` 中的 dataset 静态文件路径指向不存在的目录：

```python
dataset_dir = ROOT_DIR / "dataset"  # 不存在
```

实际 dataset 目录在 `backend/storage/image-processing/dataset/`。

### 解决方案

1. 修改 `backend/app.py` 中的静态文件目录：

```diff
- dataset_dir = ROOT_DIR / "dataset"
+ dataset_dir = BASE_DIR / "storage" / "image-processing" / "dataset"
```

2. 修改前端 `useImageProcess.ts` 使用 `image_urls` 代替 `indices_list`：

```diff
- similarImages.value = result.indices_list;
+ similarImages.value = result.image_urls;
```

## 11. Python 文件 UTF-16 编码导致 SyntaxError

### 问题描述

后端启动时报错：

```
File "D:\...\backend\app.py", line 91, in <module>
    from common.schemas import Result, ApiCode
SyntaxError: source code string cannot contain null bytes
```

### 原因

在 Windows PowerShell 中使用 `echo` 命令创建 Python 文件时，默认使用 **UTF-16 LE** 编码：

```powershell
echo "" > backend/common/__init__.py  # 错误：创建 UTF-16 编码文件
```

UTF-16 编码的特点是每个 ASCII 字符后面跟着一个 null byte (`00`)，例如：

- `#` 在 UTF-8 中是 `23`
- `#` 在 UTF-16 LE 中是 `23 00`

Python 解释器无法解析包含 null bytes 的源代码文件。

### 诊断方法

使用 PowerShell 查看文件的十六进制内容：

```powershell
[System.IO.File]::ReadAllBytes("backend\common\__init__.py") | ForEach-Object { "{0:X2}" -f $_ } | Select-Object -First 10
```

如果输出包含 `FF FE`（UTF-16 LE BOM）和交替的 `00` 字节，则确认是编码问题。

### 解决方案

使用 PowerShell 的 `[System.IO.File]::WriteAllText` 方法以 UTF-8 编码创建文件：

```powershell
[System.IO.File]::WriteAllText("backend\common\__init__.py", "# Common module`n", [System.Text.Encoding]::UTF8)
```

或者直接删除文件并用编辑器重新创建。

### 预防措施

- **不要使用** `echo` 或重定向 `>` 创建 Python 文件
- 使用代码编辑器（VS Code 等）创建文件，确保保存为 UTF-8 编码
- 如果必须用命令行创建，使用 `Set-Content -Encoding UTF8` 或 `[System.IO.File]::WriteAllText`
